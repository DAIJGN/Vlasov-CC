# 如何让 AI 完成 Coding

真正高效的 AI Coding，不是简单地让模型“帮我写代码”，而是把项目本身设计成适合 agent 理解、拆解、修改和验证的结构。也就是说，代码架构、文档架构和验证体系都要面向 agent 重新组织。

我现在采用的是一种更偏 Agent- Native / Vibe Coding 的工程设计思路：人不再只是在 prompt 里描述需求，而是把项目知识、代码边界、组合规则和验证流程沉淀成 agent 可以读取和执行的系统。

这套 coding 架构可以抽象成四层：

1. Skill 层：告诉 agent 去哪里找知识
2. Knowledge 层：沉淀可复用的工程知识
3. Primitive 层：把代码拆成可组合的原子能力
4. Verification 层：让 agent 写完以后可以自我验证

这四层的目标不是让 AI 一次性生成完美代码，而是让 AI 在一个有边界、有索引、有测试、有反馈的环境里持续迭代。

## 1. Skill 层：文档即路由

在传统项目里，README 和文档主要是给人看的；但在 Agent- Native 项目里，文档首先要能被 agent 快速消费。

因此，高层 skill 不需要重复大量内容，而是承担“路由索引”的作用：

- 这个问题属于哪个模块？
- 应该先读哪份设计文档？
- 修改代码前需要了解哪些前提？
- 相关的 primitive、combo、tuning 文档在哪里？

这种设计的关键是：高层 skill 轻量化，底层知识结构化。

高层索引只负责把 agent 引导到正确位置，真正的知识沉淀在 canonical docs 里。这样可以避免多个地方重复维护内容，也能减少 agent 读错上下文或读到过期信息的概率。

## 2. Knowledge 层：把经验写成结构化知识卡片

为了让 agent 能稳定使用文档，skill 文档不能只是自由文本，而应该有固定结构。文档通常包括：

- 类型：primitive / combo / tuning / onboarding
- 标签：memory、activation、parallel、checkpoint 等
- 依赖：使用这个能力前必须满足什么条件
- 摘要：一句话说明它解决什么问题
- 前提：什么时候可以用，什么时候不能用
- 代码入口：对应的核心文件或函数
- 相邻 skill：相关能力和替代方案
- 边界限制：不能让 agent 误用的地方

## 3. Primitive 层：代码要可组合，而不是一整块 monolith

如果代码本身是一个高度耦合的 monolith，agent 很难安全修改，因为它无法准确判断边界和副作用。

所以这套架构更强调 primitive- first：把系统拆成多个独立、可替换、可组合的原子能力。

可以抽象成三类代码：

- Runtime：对外暴露稳定 API，是上层框架或任务调用的入口。
- Primitive：真正可复用、可组合、可替换的原子积木。
- Model/Implementation：基于 primitive 组装出来的具体实现或参考方案。

这样设计以后，agent 不需要一次理解整个系统，只需要理解：

- 当前任务涉及哪个 primitive；
- 这个 primitive 的输入输出契约是什么；
- 哪些 model 或 runtime 正在消费它；
- 修改后需要跑哪些验证。

这让 AI Coding 从“修改一坨代码”变成“在明确边界里替换或组合积木”。

## 4. Combo 层：让 agent 探索组合空间

当 primitive 足够清晰之后，agent 只能做单点修改，还可以尝试组合不同能力。例如：

- 尝试不同并行策略组合；
- 对比 lite/hybrid/triton 等不同实现；
- 在多个实现之间做 benchmark，然后选择更优方案。

这也是 vibecoding 比传统 copilot 更强的地方：

AI 不只是补全代码，而是可以在已有约束下进行工程搜索。

但前提是项目必须提前给 agent 提供清晰的组合规则、依赖说明和验证入口。否则 agent 只会盲目试错，很难产生稳定结果。

## 5. Verification 层：必须有可验证闭环

AI Coding 最大的问题是不一定写的是对的。

所以这套架构必须配套验证闭环：

- 有 baseline 或 ground truth；
- 有 step- by- step 的精度对齐；
- 有 benchmark 衡量性能变化；
- 有测试或脚本检查接口是否被破坏；
- 有日志和结果记录方便复盘。

理想状态下，agent 的工作流程应该是：

1. 读取 skill，确定任务边界。
2. 找到代码入口和相关 primitive。
3. 生成或修改实现。
4. 跑 baseline 对齐或单元测试。
5. 跑 benchmark 看收益。
6. 根据结果继续修正。
7. 输出修改说明、验证结果和后续风险。

这套架构下，AI Coding 可以分成几个层级：

| 模式 | 人的角色 | Agent 的角色 | 适合任务 |
|------|----------|--------------|----------|
| Agent as Composer | 给目标和约束 | 读 skill、组装配置、生成实现、跑验证 | 新模型组合、快速搭建实验 |
| Agent 辅助 Composer | 给明确改造方向 | 读取相关 primitive，完成局部替换 | 替换某个模块、改一段实现 |
| Agent 自主 Optimize | 给优化目标 | 探索多个组合，自动 benchmark | 性能调优、并行策略尝试 |
| Agent 自主 Research | 给大方向和验证标准 | 发现瓶颈、提出假设、实现并验证 | 更开放的工程优化和研究探索 |
